# Remove wrk2 Load-Generator Support — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Remove the `wrk2` load generator from Meshery completely (code, dependency, git submodule, build wiring, UI options, docs) while keeping fortio fully functional and ensuring legacy `wrk2` data degrades gracefully to fortio.

**Architecture:** fortio becomes the sole load generator. The load-generator value becomes vestigial on the execution path (every test runs fortio), so legacy `wrk2` in profiles/preferences/requests is tolerated and transparently runs on fortio — no errors. Validation stays meaningful (genuine garbage generators still rejected) via a documented legacy-compat allow-list, which keeps the `ErrLoadgenerator` meshkit error code referenced and avoids error-registry regeneration.

**Tech Stack:** Go (server, mesheryctl/cobra), TypeScript/React + Redux Toolkit + RTK Query (ui), vitest, bats, Docker, Make, Hugo docs.

**Spec:** `docs/superpowers/specs/2026-05-18-remove-wrk2-design.md`

**Refinements discovered during planning (within the approved spec's envelope):**
- The `server/cmd/gowrk2` submodule is **orphaned** — declared in `.gitmodules` only, with no gitlink in the tree, not initialized in `.git/config`, no working-tree directory. Cleanup is just `git rm .gitmodules` (its only content). The spec's conditional `.git/config`/`.git/modules` steps are no-ops here; included defensively.
- `validators.go` / `user_handler.go` keep a **legacy-tolerant allow-list** (accept `fortio`, empty, and legacy `"wrk2"`; reject anything else) rather than deleting validation. This honors graceful fallback, keeps `ErrLoadgenerator` referenced (no meshkit error-registry churn), and preserves meaningful validation. This is the spec's own "otherwise leave it" branch for `ErrLoadgenerator`.
- The `prefTest` redux slice does not (and need not) normalize: graceful fallback is enforced upstream at the RTK Query `getLoadTestPrefs.transformResponse` boundary (the "UI normalizer"), and the form will offer fortio only. `prefTest.test.ts` wrk2 values are arbitrary payloads for reducer tests; they become `fortio` with no slice change.
- The `infoloadGenerators` JSX in `performance-helpers.tsx` only contains a wrk2 `<li>` (the "Coordinated Omission" prose lives in docs, not this file).

**Grep-gate allow-list (intentionally-preserved `wrk2` hits after completion):** release notes under `docs/content/en/project/releases/`; mesheryctl golden/fixture files under `mesheryctl/internal/cli/root/perf/{testdata,fixtures}/`; base64 image blobs in `docs/data/catalog/**/design.yml`; coincidental substrings in `ui/package-lock.json` & `install/docker-extension/ui/package-lock.json`; the deliberate legacy-compat `"wrk2"` literals + comments in `server/models/validators.go`, `server/handlers/user_handler.go`, and the legacy-fallback comments in `server/handlers/load_test_handler.go`; `server/models/validators_test.go` legacy-tolerance regression cases; this plan and the spec.
