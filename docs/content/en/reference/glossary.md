Meshery introduces several project-specific terms that are used consistently across
its documentation, codebase, and community. This page serves as a single reference
for those terms.

## Adapter

A Meshery Adapter is a gRPC-based plugin that extends Meshery's management capabilities
to a specific cloud native technology (e.g., Istio, Linkerd, Consul). Adapters translate
Meshery's API calls into technology-specific operations and report operational data back
to Meshery Server.

See: [Adapters]({{< ref "/concepts/architecture/adapters" >}})

## Broker

The Meshery Broker (based on NATS) enables asynchronous, event-driven communication
between Meshery Server and its components (such as MeshSync and Adapters). It acts as
a message bus for internal system events and telemetry.

See: [Broker]({{< ref "/concepts/architecture/broker" >}})

## Catalog

The Meshery Catalog is a library of curated, reusable cloud native design templates,
patterns, and configurations. Users can publish their own designs to the catalog or
clone published designs to use as starting points.

See: [Catalog](https://meshery.io/catalog)

## Component

A Component is the atomic unit of a Meshery Model. It represents a specific Kubernetes
resource type (e.g., `Deployment`, `Service`, `VirtualService`). Components carry
schema-validated configuration and are used within Designs.

See: [Components]({{< ref "/concepts/logical/components" >}})

## Conformance

Meshery provides built-in conformance testing to validate whether a service mesh
adheres to industry standards such as SMI (Service Mesh Interface). Results indicate
which specification features a given service mesh supports, helping teams make
informed technology decisions.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## Connection

A Connection represents an authenticated link between Meshery and an external system
(such as a Kubernetes cluster, Prometheus instance, or cloud provider). Connections
are managed and can be associated with Environments.

See: [Connections]({{< ref "/concepts/logical/connections" >}})

## Credential

A Credential stores the authentication data (tokens, kubeconfigs, API keys) required
to establish a Connection to an external system.

See: [Credentials]({{< ref "/concepts/logical/credentials" >}})

## Database

Meshery Server uses an internal SQLite database to persist registered Models,
Components, Connections, Credentials, Environments, Designs, and performance results.
The database is stored locally at `~/.meshery/config/` and is managed automatically
by Meshery Server.

See: [Database]({{< ref "/concepts/architecture/database" >}})

## Design

A Meshery Design is a collection of configured Components and their Relationships,
representing a desired state of cloud native infrastructure. Designs are the primary
artifact for GitOps workflows in Meshery and can be deployed, undeployed, exported,
or shared via the Catalog.

See: [Designs]({{< ref "/concepts/logical/designs" >}})

## Docker Extension

The Meshery Docker Extension integrates Meshery directly into Docker Desktop,
providing access to visual design, lifecycle management, and performance testing
without requiring a separate Kubernetes cluster or browser tab.

See: [Docker Extension]({{< ref "/installation/docker/docker-extension" >}})

## Dry Run

A Dry Run simulates applying a Meshery Design to a target cluster to check for
configuration errors, resource conflicts, and policy violations — without making
any actual changes. Meshery uses Kubernetes' built-in dry-run mechanisms internally.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## Environment

An Environment is a named grouping of Connections and Credentials. Environments make
it easier to manage and share sets of infrastructure resources (e.g., `production` or
`staging`) across teams and Workspaces.

See: [Environments]({{< ref "/concepts/logical/environments" >}})

## Event

An Event is a structured record of an occurrence within Meshery, generated by user
actions or automatically by the system. Events have an Action, Category, Severity,
and Metadata, and are surfaced to users through the Notification Center.

See: [Architecture]({{< ref "/concepts/architecture" >}})

## Filter

A Filter (also called a WASM Filter) is a WebAssembly binary that extends the data
plane of an Envoy-based service mesh proxy with custom logic — such as authentication
or request transformation — without recompiling the proxy. Meshery manages filters
via `mesheryctl filter`.

See: [mesheryctl filter]({{< ref "/reference/mesheryctl/filter" >}})

## GitOps

GitOps is an infrastructure management practice where desired infrastructure state
is defined declaratively in Git and reconciled automatically. Meshery provides
GitOps-native workflows through Kanvas, enabling teams to design, version, and
deploy cloud native infrastructure using Git as the source of truth.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## Grafana

Grafana is an open-source metrics visualization platform integrated with Meshery.
It is used to display performance test results and infrastructure metrics as
dashboards. Meshery supports both Static Boards (built-in) and Dynamic Boards
(user-defined via Grafana SDK).

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## Greenfield and Brownfield

**Greenfield** refers to new infrastructure being designed and deployed from scratch
using Meshery's canvas. **Brownfield** refers to existing infrastructure that is
discovered, imported, and managed by Meshery — typically via MeshSync's automatic
cluster discovery.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## Helm

Helm is the Kubernetes package manager. Meshery treats Helm charts as first-class
citizens, supporting import, dry-run validation, and direct deployment of Helm
charts as Meshery Designs. Meshery itself can also be installed via a Helm chart.

See: [Install with Helm]({{< ref "/installation/kubernetes/helm" >}})

## Integration

An Integration represents any supported cloud native technology that Meshery can
connect to, model, or manage — including Kubernetes distributions, cloud providers,
service meshes, and observability tools. Meshery ships with 380+ integrations, each
represented as a Model in the Registry.

See: [Integrations](https://meshery.io/integrations)

## Kanvas

Kanvas is the visual design and management extension for Meshery. It provides a
drag-and-drop canvas interface for creating, editing, and collaborating on Meshery
Designs in real time. Kanvas has two primary modes: Designer and Visualizer.

See: [Kanvas]({{< ref "/extensions/kanvas" >}})

## Kanvas Designer

Kanvas Designer is the visual "diagram as code" mode within Kanvas. It provides a
drag-and-drop canvas for creating Meshery Designs, importing existing configurations
(Helm charts, Kustomize overlays, Docker Compose files), and collaborating with
teammates in real time.

See: [Kanvas]({{< ref "/extensions/kanvas" >}})

## Kanvas Visualizer

Kanvas Visualizer is the live infrastructure management mode within Kanvas. It
displays a real-time topology of resources discovered by MeshSync inside a connected
Kubernetes cluster, showing the actual running state of infrastructure rather than
its desired design state.

See: [Kanvas]({{< ref "/extensions/kanvas" >}})

## Kustomize

Kustomize is a Kubernetes-native configuration management tool that applies
template-free customizations to YAML manifests. Meshery supports importing and
managing Kustomize configurations as Meshery Designs.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## Lifecycle

Lifecycle refers to the full management span of cloud native infrastructure across
three phases: **Day 0** (provisioning), **Day 1** (initial configuration and
deployment), and **Day 2** (ongoing operation, monitoring, and upgrades). Meshery
provides tooling for all three phases.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## Load Generator

A Load Generator is a tool used by Meshery to produce synthetic HTTP traffic for
performance testing. Meshery supports three generators: **Fortio** (Go-based,
simple QPS targeting), **wrk2** (avoids coordinated omission for accurate latency),
and **Nighthawk** (C++, distributed, Layer 7).

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## Load Test

A Load Test is a performance evaluation that generates simulated traffic against a
target endpoint to measure latency, throughput (RPS), and error rates. Meshery
abstracts multiple load generators behind a single unified interface.

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## Local Provider

The Local Provider (shown as **"None"** in the UI) is a built-in Meshery provider
that runs entirely within your local instance. It requires no external authentication
and stores all data locally. It is intended for single-user, ephemeral use and does
not support multi-user collaboration.

See: [Architecture]({{< ref "/concepts/architecture" >}})

## meshconfig

`meshconfig` is the configuration file used by `mesheryctl`, stored by default at
`~/.meshery/config.yaml`. It manages cluster contexts, provider settings, and access
configuration — conceptually similar to `kubeconfig` for Kubernetes.

See: [mesheryctl Reference]({{< ref "/reference/mesheryctl" >}})

## Meshery Cloud

Meshery Cloud (`cloud.meshery.io`) is the managed SaaS platform and canonical Remote
Provider for Meshery. It provides identity management, multi-user collaboration,
persistent storage, private catalogs, GitOps integration, and RBAC across multiple
Meshery instances.

See: [Meshery Cloud](https://cloud.meshery.io)

## mesheryctl

`mesheryctl` is the official command-line interface for Meshery. It manages the full
Meshery lifecycle — starting and stopping Meshery, managing connections, running
performance tests, interacting with the registry, and managing designs — all from
a terminal, analogous to `kubectl` for Kubernetes.

See: [mesheryctl Reference]({{< ref "/reference/mesheryctl" >}})

## MeshMap

MeshMap is the former name of Kanvas, Meshery's visual infrastructure design and
management extension. The product has been rebranded to Kanvas. Older documentation
and community discussions may still reference MeshMap.

See: [Kanvas]({{< ref "/extensions/kanvas" >}})

## MeshMark

MeshMark is a universal performance index produced by Meshery using the SMP
specification. It provides a normalized score that allows organizations to gauge
and compare their service mesh efficiency across different deployments and cloud
environments.

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## MeshSync

MeshSync is a Kubernetes operator component of Meshery that continuously discovers
and synchronizes the state of Kubernetes resources into Meshery's internal database.
It acts as the "eyes" of Meshery inside a cluster.

See: [MeshSync]({{< ref "/concepts/architecture/meshsync" >}})

## Model

A Meshery Model is a versioned, schema-validated package that describes a cloud native
technology (e.g., Kubernetes, Istio, AWS). A Model contains one or more Components and
defines the Relationships between them. Models are registered in the Meshery Registry.

See: [Models]({{< ref "/concepts/logical/models" >}})

## NATS

NATS is the open-source messaging system used by the Meshery Broker to enable
event-driven, asynchronous data streaming between MeshSync (inside a Kubernetes
cluster) and the Meshery Server (which may run outside the cluster).

See: [Broker]({{< ref "/concepts/architecture/broker" >}})

## Namespace

A Namespace is a Kubernetes mechanism for isolating groups of resources within a
cluster. Meshery leverages Kubernetes namespaces when deploying Designs and managing
multi-tenant environments.

## Notification

A Notification is the user-facing display of a system Event in the Meshery UI's
Notification Center. Notifications show read/unread status, urgency level, and
source, informing users of deployment outcomes, policy violations, and connection
status changes.

See: [Architecture]({{< ref "/concepts/architecture" >}})

## OCI (Open Container Initiative)

OCI is a standard for container image formats and registries. Meshery Models can be
packaged as OCI-compatible images and distributed via any OCI-compliant registry
(Docker Hub, GitHub Container Registry, Amazon ECR) using the ORAS toolchain.

See: [Models]({{< ref "/concepts/logical/models" >}})

## OPA (Open Policy Agent)

OPA is the open-source policy engine integrated into Meshery's policy enforcement
system. Combined with the Rego query language, OPA evaluates infrastructure Designs
against governance rules before and during deployment.

See: [Policies]({{< ref "/concepts/logical/policies" >}})

## Operator

The Meshery Operator is a Kubernetes operator that manages the lifecycle of Meshery's
in-cluster components, including MeshSync and the Broker.

See: [Operator]({{< ref "/concepts/architecture/operator" >}})

## Organization

An Organization is the top-level administrative entity in Meshery Cloud. It represents
a company or business unit and acts as the root container for all resources —
Workspaces, Teams, Designs, Environments, and Connections.

See: [Workspaces]({{< ref "/concepts/logical/workspaces" >}})

## Pattern

A Pattern is a reusable, composable unit of cloud native infrastructure configuration.
Patterns are derived from industry best practices and are surfaced through the Meshery
Catalog.

See: [Patterns]({{< ref "/concepts/logical/patterns" >}})

## Performance Profile

A Performance Profile is a saved, reusable configuration for running load tests in
Meshery. It stores test parameters — target URL, QPS, duration, concurrency, and load
generator choice — so identical tests can be repeated for consistent benchmarking.

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## Playground

The Meshery Playground (`play.meshery.io`) is a hosted, ephemeral Meshery instance
provided by Layer5. It allows users to explore Meshery features — Kanvas, Designs,
the Registry, and live cluster management — without any local installation.

See: [Meshery Playground]({{< ref "/installation/playground" >}})

## Policy

A Policy in Meshery is an OPA (Open Policy Agent) Rego rule used to evaluate and
enforce configuration constraints on Designs. Policies can be context-aware and operate
across components and their relationships.

See: [Policies]({{< ref "/concepts/logical/policies" >}})

## Prometheus

Prometheus is an open-source metrics collection and monitoring system integrated with
Meshery. Meshery queries Prometheus for infrastructure and performance metrics during
load tests and supports auto-discovery of Prometheus instances within connected clusters.

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## RBAC (Role-Based Access Control)

RBAC is Meshery's authorization framework for controlling access to platform resources
and actions. Permissions are grouped into Keychains (sets of Permission Keys) assigned
to Roles, which are then associated with users at the Organization, Team, or Workspace
level.

See: [Workspaces]({{< ref "/concepts/logical/workspaces" >}})

## Registry

The Meshery Registry is the internal database that stores all registered Models,
Components, and Relationships. The `mesheryctl registry` commands interact with it.

See: [Registry]({{< ref "/concepts/logical/registry" >}})

## Rego

Rego is the declarative query language used by OPA. Meshery uses Rego to write
governance policies that are evaluated before and during the deployment of Designs,
enabling context-aware constraint enforcement across components and their relationships.

See: [Policies]({{< ref "/concepts/logical/policies" >}})

## Relationship

A Relationship describes how two or more Components interact with each other within
a Design (e.g., a Pod mounting a PersistentVolume, a Service selecting a Deployment).
Relationships drive visual connections in Kanvas and power policy evaluation.

See: [Relationships]({{< ref "/concepts/logical/relationships" >}})

## Remote Provider

A Remote Provider connects a local Meshery instance to an external platform such as
Meshery Cloud (`cloud.meshery.io`). It enforces user authentication, enables
multi-user collaboration, persists designs and performance history, and unlocks
additional extensions such as Kanvas.

See: [Architecture]({{< ref "/concepts/architecture" >}})

## Server

Meshery Server is the central control-plane component of Meshery. It exposes REST
and GraphQL APIs, coordinates Adapters via gRPC, manages the internal database,
handles authentication through providers, and serves the Meshery UI.

See: [Architecture]({{< ref "/concepts/architecture" >}})

## Service Mesh

A Service Mesh is a dedicated infrastructure layer for managing service-to-service
communication within microservices applications. It provides traffic management,
mutual TLS, observability, and resilience policies without changing application code.
Meshery was originally built to manage service meshes (e.g., Istio, Linkerd, Consul)
and has since expanded to manage all cloud native infrastructure.

See: [Adapters]({{< ref "/concepts/architecture/adapters" >}})

## SMI (Service Mesh Interface)

SMI is a vendor-neutral open standard defining a common API for service meshes on
Kubernetes. Meshery provides built-in SMI conformance testing to validate whether a
given service mesh correctly implements the SMI specification.

See: [Infrastructure Management]({{< ref "/guides/infrastructure-management" >}})

## SMP (Service Mesh Performance)

SMP is a vendor-neutral CNCF specification for consistently measuring and comparing
service mesh performance. Meshery is the canonical implementation of SMP and uses it
to generate MeshMark scores, enabling apples-to-apples performance comparisons across
deployments.

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## Snapshot

A Snapshot is a point-in-time visual representation of infrastructure state — such as
a Kubernetes cluster topology or a Helm chart. Snapshots can be embedded into GitHub
Pull Requests via the Kanvas Snapshot GitHub Action for visual review of infrastructure
changes before merge.

See: [Kanvas]({{< ref "/extensions/kanvas" >}})

## Team

A Team is a group of users within an Organization. Teams are used to assign access
permissions to Workspaces, Environments, and Designs, enabling role-based collaboration
across projects within the same Organization.

See: [Workspaces]({{< ref "/concepts/logical/workspaces" >}})

## Telemetry

Telemetry in Meshery refers to the collection, storage, and visualization of
infrastructure and performance metrics. Meshery integrates with Prometheus for metrics
collection and Grafana for dashboard visualization across managed clusters and service
mesh deployments.

See: [Performance Management]({{< ref "/guides/performance-management" >}})

## Token

A Token is the JWT-based authentication credential that authorizes access to Meshery's
REST and GraphQL APIs when using a Remote Provider. Tokens are stored locally at
`~/.meshery/auth.json` and obtained via `mesheryctl system login`.

See: [mesheryctl system login]({{< ref "/reference/mesheryctl/system/login" >}})

## User

A User is an individual with a Meshery account. Users can belong to multiple
Organizations and Teams, and their access to resources (Designs, Environments,
Connections) is governed by RBAC roles assigned within those Teams and Organizations.

## View

A View in Kanvas is a saved, named perspective of a Meshery Design canvas. Views let
teams focus on meaningful subsets of components (e.g., a "networking view" or a
"storage view") without duplicating the underlying design.

See: [Kanvas]({{< ref "/extensions/kanvas" >}})

## Workspace

A Workspace is the top-level organizational unit in Meshery for teams. It groups
related Designs, Environments, and Connections together and controls access for
team members.

See: [Workspaces]({{< ref "/concepts/logical/workspaces" >}})
