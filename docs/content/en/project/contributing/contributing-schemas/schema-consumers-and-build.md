This page covers the downstream artifacts produced by `make build` and how each consumer — Go, TypeScript, and the RTK Query layer — uses them.

## Build pipeline {#build-pipeline}

```
schemas/constructs/**/*.yaml, api.yml   ← you write these
          │
          ▼  make bundle-openapi
    _openapi_build/
    ├── merged_openapi.yml
    ├── cloud_openapi.yml      ← x-internal: ["cloud"] ops only
    └── meshery_openapi.yml    ← x-internal: ["meshery"] ops only
          │
          ├─▶ make generate-golang
          │     └── models/v1beta1/<construct>/<construct>.go
          │
          ├─▶ make generate-ts
          │     └── typescript/generated/v1beta1/<Construct>/
          │         ├── <Construct>.d.ts       (TypeScript types)
          │         └── <Construct>Schema.ts   (schema as const JS object)
          │
          └─▶ make generate-rtk
                └── typescript/rtk/           (RTK Query hooks)
```

## Makefile targets {#makefile-targets}

| Target | What it does |
|---|---|
| `make setup` | `go mod tidy` + `npm install` |
| `make build` | Full build: validate + bundle + Go + TypeScript + RTK + tests |
| `make bundle-openapi` | Resolve `$ref` and merge specs into `_openapi_build/` |
| `make generate-golang` | Generate Go structs (requires `bundle-openapi` first) |
| `make generate-ts` | Generate TypeScript types and schema objects |
| `make generate-rtk` | Generate RTK Query client hooks |
| `make validate-schemas` | Run blocking validation rules (CI gate) |
| `make validate-schemas-strict` | Run all 42 rules including advisory |
| `make audit-schemas` | Advisory report — new issues only vs baseline |
| `make audit-schemas-full` | Full advisory backlog |
| `make consumer-audit` | Check downstream consumers for casing conformance |
| `npm run build` | Build TypeScript distribution (`dist/`) |
| `go test ./...` | Run all Go tests |

## What NOT to commit {#do-not-commit}

| Path | Generated by | Do not commit |
|---|---|---|
| `models/<version>/<pkg>/<pkg>.go` | `make generate-golang` | Yes |
| `typescript/generated/` | `make generate-ts` | Yes |
| `dist/` | `npm run build` | Yes |
| `_openapi_build/` | `make bundle-openapi` | Yes |

| Path | Manually maintained | Commit |
|---|---|---|
| `schemas/constructs/**/api.yml` | Author | Yes |
| `schemas/constructs/**/<construct>.yaml` | Author | Yes |
| `schemas/constructs/**/templates/` | Author | Yes |
| `typescript/index.ts` | Author | Yes |
| `models/<version>/<pkg>/<pkg>_helper.go` | Author | Yes |
| `models/core/datatype_*.go` | Author | Yes |

## Go consumers {#go-consumers}

### Generated structs {#generated-structs}

`make generate-golang` runs oapi-codegen against the bundled spec and writes Go struct files to `models/v1beta1/<construct>/<construct>.go`. The generation pipeline has several post-processing steps:

1. Add `yaml:` tags mirroring the `json:` tag value
2. Merge `x-oapi-codegen-extra-tags` entries (`db`, `gorm`, etc.) into struct tag strings
3. Validate that every `db:` tag declared in the schema is present in the output
4. Validate that every `json:` tag in the output matches the corresponding schema property name
5. Generate `zz_generated.helpers.go` for types marked with `x-generate-db-helpers: true`

### Helper files {#helper-files}

Manually maintained helper files live alongside the generated struct. Always begin with `// This is not autogenerated.`:

```
models/v1beta1/environment/
├── environment.go          # Auto-generated — DO NOT edit
└── environment_helper.go   # Manual — Entity interface, TableName(), SQL drivers
```

#### Entity interface {#entity-interface}

For constructs that need GORM database CRUD operations:

```go
// environment_helper.go
// This is not autogenerated.
package environment

import (
    "github.com/gofrs/uuid"
    "github.com/meshery/meshkit/database"
    "github.com/meshery/meshkit/models/meshmodel/entity"
    "gorm.io/gorm/clause"
)

func (e Environment) TableName() string { return "environments" }
func (e Environment) Type() entity.EntityType { return entity.Environment }
func (e *Environment) GenerateID() (uuid.UUID, error) { return uuid.NewV4() }
func (e Environment) GetID() uuid.UUID { return e.Id }

func (e *Environment) Create(db *database.Handler, hostID uuid.UUID) (uuid.UUID, error) {
    e.Id, _ = e.GenerateID()
    return e.Id, db.Omit(clause.Associations).Create(e).Error
}
```

#### SQL driver methods {#sql-drivers}

For types stored as JSON blobs in a single column, prefer `x-generate-db-helpers: true` on the schema component to auto-generate `Scan()`/`Value()`. When writing manually:

```go
// This is not autogenerated.
package mypackage

import (
    "database/sql/driver"
    "github.com/meshery/schemas/models/core"
)

// Value — always marshals; never returns (nil, nil)
func (m MyType) Value() (driver.Value, error) {
    b, err := json.Marshal(m)
    if err != nil {
        return nil, err
    }
    return string(b), nil
}

// Scan — zero the receiver on nil src
func (m *MyType) Scan(src interface{}) error {
    if src == nil {
        *m = MyType{}  // zero the receiver — do not return early with stale data
        return nil
    }
    mapVal := core.Map{}
    if err := mapVal.Scan(src); err != nil {
        return err
    }
    return core.MapToStruct(mapVal, m)
}
```

#### Core utility types {#core-types}

The `models/core/` package provides reusable types with built-in SQL compatibility:

| Type | Use case |
|---|---|
| `core.Map` | `map[string]any` with SQL driver methods — use for amorphous `metadata` fields |
| `core.NullTime` | Nullable timestamp — use for `deleted_at` |
| `core.Time` | Required timestamp — use for `created_at`, `updated_at` |

When using `core.NullTime` for a nullable timestamp field, the JSON tag is camelCase while the GORM column tag retains the snake_case DB column name — the ORM layer is the only translation boundary:

```go
import "github.com/meshery/schemas/models/core"

// For nullable timestamps (e.g., deleted_at)
// Per the canonical naming contract, the JSON tag is camelCase
// (`deletedAt`) while the GORM column tag retains the snake_case
// DB column name (`deleted_at`). The ORM layer is the only translation boundary.
type MyStruct struct {
    DeletedAt core.NullTime `json:"deletedAt" gorm:"column:deleted_at"`
}
```

In schemas, reference core types with `x-go-type-import` pointing to `models/core`:

```yaml
metadata:
  type: object
  description: Additional metadata.
  x-go-type: core.Map
  x-go-type-skip-optional-pointer: true
  x-go-type-import:
    path: github.com/meshery/schemas/models/core
    name: core
  x-oapi-codegen-extra-tags:
    db: metadata
```

## TypeScript consumers {#typescript-consumers}

### Generated files {#generated-ts-files}

`make generate-ts` produces two files per construct in `typescript/generated/v1beta1/<Construct>/`:

- `<Construct>.d.ts` — TypeScript interface definitions (types only, no runtime value)
- `<Construct>Schema.ts` — the OpenAPI schema as a const JavaScript object (runtime value for validation)

### `typescript/index.ts` — the public API surface {#index-ts}

This file is **manually maintained**. It defines what is exported from `@meshery/schemas`. When adding a new construct, add three items:

```typescript
// 1. Components import — NO .d.ts extension in the path
import { components as KeychainComponents } from "./generated/v1beta1/keychain/Keychain";

// 2. Schema import (runtime object for validation)
import KeychainDefinitionV1Beta1OpenApiSchema from "./generated/v1beta1/keychain/KeychainSchema";

// 3. Type namespace export
export namespace v1beta1 {
  // ...existing exports...
  export type Keychain = KeychainComponents["schemas"]["Keychain"];
  export type KeychainPayload = KeychainComponents["schemas"]["KeychainPayload"];
}

// 4. Schema object export
export { KeychainDefinitionV1Beta1OpenApiSchema };
```

Property names in the generated TypeScript are **verbatim from the schema** — no casing transformation. If the schema has `organization_id`, TypeScript has `organization_id`:

```typescript
import { v1beta1 } from "@meshery/schemas";

const env: v1beta1.Environment = {
  id: "...",
  name: "Production",
  organization_id: "...",   // snake_case — matches the DB-backed schema property
  schemaVersion: "...",     // camelCase — matches the non-DB schema property
};
```

### v1beta2 namespace {#v1beta2}

Constructs migrated to v1beta2 are exported under a separate namespace:

```typescript
import { v1beta2 } from "@meshery/schemas";

const design: v1beta2.Design = { ... };
const connection: v1beta2.Connection = { ... };
```

Always consume the latest canonical version. Do not import from `v1beta1` constructs that have a `v1beta2` equivalent in new code.

### Using schemas for runtime validation {#runtime-validation}

The `*Schema.ts` exports are runtime JSON Schema objects that can be used with an AJV-compatible validator:

```typescript
import { EnvironmentDefinitionV1Beta1OpenApiSchema } from "@meshery/schemas";

const isValid = ajv.validate(
  EnvironmentDefinitionV1Beta1OpenApiSchema.components.schemas.Environment,
  data
);
```

### Using RTK Query hooks {#rtk-hooks}

RTK Query clients in `typescript/rtk/` provide typed `useGetEnvironmentsQuery`, `useCreateEnvironmentMutation`, etc. hooks:

```typescript
import { mesheryApi } from "@meshery/schemas/dist/mesheryApi";

const { data, isLoading } = mesheryApi.useGetEnvironmentsQuery({ page: 0, pagesize: 10 });
```

## Testing {#testing}

### Go tests

```bash
go test ./...       # all Go tests
go build ./...      # compile check
```

### TypeScript tests

```bash
npm run build       # type check + build distribution
```

### Verifying generated output

After `make build`, spot-check:

```bash
# Check Go struct has expected db tags
grep "db:\"created_at\"" models/v1beta1/environment/environment.go

# Check TypeScript has expected property names
grep "organization_id" typescript/generated/v1beta1/environment/Environment.d.ts
```
